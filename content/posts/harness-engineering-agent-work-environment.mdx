## 1. 开篇场景：Coding Agent 漂移的那一刻

先讲一个几乎所有用过 Coding Agent 的工程师都遇过的场景。你打开 Claude Code 或者 Codex，把一个跨模块的小重构任务丢给它——"把所有用旧版 logger 的地方都换成新版 structured logger"。前 15 分钟它干得漂亮：扫仓库、列出所有文件、改了第一个、跑测试、绿了。你站起来去倒咖啡，回来发现它已经改完 12 个文件，正在准备 commit。心情大好。

然后转折来了。第 13 个文件改完，它突然在另一个根本不需要动的模块里 import 了一个不存在的 helper；第 14 个文件改完，它开始反复修改同一行代码——加注释、删注释、再加上略微不同的注释；第 15 个文件，它把一个早就在第 3 步处理过的旧 logger 重新引入回来。你点开它的执行日志，发现 token 已经烧掉了一大半，进度反而开始倒退。

你拍着桌子在想：到底是模型不行，还是我"用模型"的方式不行？

这个瞬间——技术圈里有一个准确的词叫做 **"漂移"（drift）**——是 Harness 工程登场的真正起点。漂移不是模型在某一步算错了，而是它**在长时间执行里慢慢偏离了原本的任务方向**，最终走到一个没人意料的地方。漂移不会被换更聪明的模型彻底治好——GPT-5 比 GPT-4 漂移得晚一点、但终究会漂；它需要的不是更强的脑子，而是**一套围绕模型的"工作环境"**。Harness 工程，本质就是研究怎么搭这套环境。

这一篇文章会从这个具体场景出发，一路向下追问到模型本质——为什么会漂移、漂移有几种成因、每种成因对应什么样的"缰绳"。读完之后，你下次再看到 Agent 抽风，应该能立刻反应过来"哦，这是哪一层环境没设计好"，而不是单纯地骂模型。

## 2. 第一性原理：模型是函数，不是程序员

要理解漂移，先得把"模型是什么"这件事彻底搞清楚。大多数人在用 ChatGPT 一段时间之后会下意识把它当作"一个会写代码的同事"——这其实是个非常贵的误解。**模型本质上不是同事，它是一个无状态的概率函数**，长这个样子：

```text
P(next_token | context_window)

```

每次你看到它"思考"、"决策"、"规划"，背后发生的事情其实只有一件——**它读完你塞进上下文窗口里的所有 token，然后输出一个最可能的下一个 token**。仅此而已。它没有"我是谁"的稳定锚点，没有"我刚才在做什么"的内在记忆，没有"我接下来要做什么"的内置规划。这些看起来像意识的东西，全部依赖一件事——**上下文窗口里有没有相应的信息**。

这个第一性原理一旦装进脑子，就能瞬间解释 Agent 的几乎所有诡异行为。**为什么它会在第 50 步突然忘了第 5 步的决定？**因为窗口太长，关键信息被新内容挤到了"长尾位置"，模型注意力分配不到。**为什么它会反复来回改同一行？**因为它的上下文里没有"我刚才已经改过这里"的强信号，每一步独立判断都会得出"这里需要再调整"的结论。**为什么它会编造不存在的函数？**因为模型本质是生成最可能的 token——当上下文没有清晰提示函数清单时，统计上"看起来很像存在"的函数名会被高分采样。

把这件事用一个工程师熟悉的类比来表达——**模型不是一个长期雇员，而是一个每次都失忆、只能根据眼前桌上摊着的资料做判断的临时工**。它的每一次"决定"都是一次零起点的判断；它桌上摊着什么资料、资料怎么摆放、有没有被打过红圈，决定了这次判断的质量。**Harness 工程，归根结底就是研究"桌上的资料怎么摆"这件事**——只不过摆的不只是文字，还有规则、工具、节奏和反馈。

## 3. 三代范式真正解决的，是同一个问题——漂移

有了"模型是函数"这个原理，再回头看 Prompt → Context → Harness 三代范式，会发现它们其实在解决**同一个问题在不同时间尺度上的版本**——漂移。

**Prompt Engineering 时代解决的是"单次 turn 内的漂移"**。模型读到你的问题之后，可能朝十几个方向理解；用 Few-shot 给两个示例、用 Chain-of-Thought 提示"先想清楚再回答"、用角色定义把语境锁死——这些都是在**一次对话内、几秒钟尺度上**把模型的输出钉到你想要的方向。

**Context Engineering 时代解决的是"多 turn 之间的漂移"**。当你接一个 RAG 系统，模型每次输出依赖召回的资料；当你接一个 Agent 框架，模型每次 tool call 之后要把结果消化进上下文——这一阶段的漂移来自"塞进窗口的资料和此刻任务的相关性正在下降"。Context Engineering 就是研究怎么**在分钟尺度上让窗口内容始终对齐当前任务**：怎么召回、怎么排序、怎么压缩、怎么扔掉旧的。

**Harness Engineering 时代解决的是"长任务执行里的漂移"**。当 Agent 跑一个跨小时的任务——比如一次百文件重构、一次端到端 bug 修复、一次 5 轮自动 review——前面那些机制还是会失效。原因是**没有任何一份资料能预先涵盖"接下来几个小时里所有可能出现的局面"**，所以你不能光靠"塞资料"。你必须让模型在跑的过程中持续被"约束、反馈、纠偏"——这是系统层面的工作，需要 Linter、钩子、状态机、规则文件协同作业。

把这三层串起来看，**每一代范式都是在解决前一代留下来的"更长时间尺度上的漂移"**。这不是替代关系，是叠加关系——Harness 系统里依然要写 Prompt、依然要管 Context，只不过它们成了更大机器里的零件。明白了这个关系，下面所有原则的"为什么"就都通了。

## 4. 把 Agent 当新员工：理解 Harness 最直观的类比

"模型是函数"这个原理对工程师友好，但讲给团队里非技术同学就不太通。这里给一个更直观的类比，几乎能解释 Harness 工程的每一条原则——**把 Agent 当成一个刚入职、聪明但完全没有公司背景知识的新员工**。

这个新员工的特点是：智商在线（毕竟是上百亿参数训出来的）、英文中文都好、各种通用知识都懂，但**他对你公司的业务一无所知、对你团队的规矩一无所知、对你代码库的历史一无所知**。他能力很强，但需要 onboarding。如果你 onboarding 做得好，他第二周就能独立产出；做得不好，他第二个月还在反复问"这个接口为什么这么设计、那个字段是什么意思"。

当你把这个画面装进脑子，三代范式立刻有了对应物——

- **Prompt** = 你布置给他的**单次任务说明**。说清楚要做什么、做到什么程度、有什么注意事项。说得越清晰他做得越好。
- **Context** = 你递给他的**背景资料**。这个项目以前怎么干的、相关代码在哪里、上下游怎么联动。资料越齐他理解越准。
- **Harness** = 你团队的**整套工程文化**。代码规范、CI、code review、postmortem、入职手册、模块 README、Slack 频道、内部 wiki——一整套让"任何新人来都能稳定产出"的支撑系统。

这个类比一旦立住，所有 Harness 原则都能秒懂：

- **"代码仓库即唯一事实源"**——因为新员工不在你 Slack 里、也没见过你的口头传统，只能读文档。
- **"AGENTS.md 要分层"**——给他一份 500 页的总手册他根本读不完；给他一份 5 页的目录加各模块 README，他能按需检索。
- **"Linter 报错要解释为什么"**——你不能要求他每次违规都来问你；报错本身就是一次微型的 onboarding。
- **"用约束换自主"**——团队规矩越清楚，新员工越敢自己拍板，因为他知道边界在哪。

更妙的是，这个类比还预测了一个反直觉的结论——**把 Agent 用好的团队，往往同时也是新人 onboarding 做得最好的团队**。因为给 Agent 看的资料，本质上和给新员工看的资料是同一份。OpenAI 内部那些"为 Codex Agent 写 AGENTS.md"的工程师后来都反馈：自从这件事做完，新员工上手时间从两个月压缩到两周。一份资料，两份红利——这才是 Harness 工程最被低估的附带价值。

## 5. Spiky Intelligence 的根源：训练分布决定能力分布

讲完原理和类比，再回头看那个最经典的 Harness 现象——**Spiky Intelligence（尖刺状智能）**。模型在不同任务上的表现极度不均匀：有些做得比中级工程师还稳，有些莫名其妙地差。这种"高低起伏的尖刺"不是 bug，是结构性的，理解它的根源才知道 Harness 该补在哪里。

根源就一句话——**模型的能力分布完全由训练数据分布决定**。React 写得溜，因为开源仓库里 React 代码以 PB 计；你公司内部那个三年前用 vibe-driven design 出来的微服务框架写得拉胯，因为训练集里压根没有这个东西。Leetcode 题做得漂亮，因为题解满网都是；你团队的鉴权流程它一塌糊涂，因为这只活在内部 wiki 和老员工脑子里。**能力的"高峰"和"低谷"，对应的是训练数据的"密集区"和"真空区"**。

这个观察推出一个非常重要的工程结论——**你的 Harness 不是要让模型变聪明，而是要在"训练分布真空区"那一片为它补足知识、装上护栏**。具体来说：

| 能力区域 | 模型表现 | Harness 该补什么 |
|-|-|-|
| 训练密集区（React、Python、SQL） | 强、稳定 | 少干预，让它自由发挥 |
| 训练稀疏区（你公司业务术语） | 会蒙对但容易漂 | 用 AGENTS.md / RAG 补领域知识 |
| 训练真空区（你内部框架） | 纯靠编 | 用 Linter + 错误模板把规则注入 |
| 反模式区（已废弃的旧 API） | 仍然会用 | 用钩子拦截 + 报错指引迁移 |

看懂这张表之后，**Harness 的设计就有了清晰的目标函数——找出你项目里所有"训练分布外的尖刺低谷"，逐个补成平地**。不需要全面铺开，先补最常被踩的几个谷。这是一种非常 ROI-friendly 的工程节奏，也是 OpenAI 那帮人能在 5 个月内驯服百万行代码任务的关键——他们没有试图给 Agent "全面赋能"，只是非常专注地把自家代码库的几个深谷给填了。

## 6. Harness 的五件套：从输入到执行的全链路控制

讲完"为什么需要 Harness"，开始讲"Harness 长什么样"。一个完整的 Harness 系统由五个相互耦合的组件组成，我把它们按"模型每跑一步会用到什么"的顺序串起来——这样比单独列名更容易记。

想象一次 tool call 的完整生命周期：模型读上下文 → 决定调什么工具 → 工具执行 → 结果回填上下文 → 模型继续下一步。**五件套就分别覆盖这条生命周期的五个关键节点**。

> **① 系统提示（System Prompt）**——生命周期最开头那个永不消失的"任务身份卡"。它告诉模型"你是谁、你在做什么、你绝不能做什么"。这是 Prompt Engineering 的遗产，但放到 Agent 系统里，它不再是"对话第一句话"，而是**每一次 turn 都会被重新加载的系统宪章**，所以必须短到不挤占工作记忆，又精确到能在任何漂移瞬间把模型拉回来。
> **② 工具定义（Tool Schema）**——决定模型"能做什么"的硬边界。每个工具的输入输出格式、副作用范围、调用代价都写在 schema 里。一个常被忽略的真相是**工具不是越多越好**——给 Agent 一把瑞士军刀，它会在选刀片上耗掉一半 token；给它三把单刀，反而干净利落。
> **③ 中间件钩子（Middleware Hooks）**——在每次 tool call 前后插入的拦截层。pre-commit 钩子让代码必须过 Linter 才能写入，post-tool 钩子把超长命令输出截断到合理大小，context-trim 钩子在窗口快满时自动 summarize。**这是真正让 Agent "可控"的关键层——它把模型概率性的输出，变成确定性的边界内行为**。
> **④ 执行流程（Execution Flow）**——Agent 主循环的节奏控制器。什么时候允许它跑下一步、什么时候必须停下来等用户确认、什么时候触发自我反思、什么时候强制结束。这一层有点像状态机，决定了 Agent 在长任务里走得快还是稳。
> **⑤ 上下文管理策略（Context Strategy）**——窗口里每一时刻应该装什么的策略对象。这是 Context Engineering 的衣钵，但被升级成了"可配置、可观测、可调试"的一等公民——你可以打开它看每一步窗口里塞了什么、谁挤掉了谁。

这五件套之间不是并列关系而是**互相耦合的系统**。系统提示给"身份"，工具定义给"动作"，中间件钩子给"安全网"，执行流程给"节奏"，上下文管理给"视野"。任何一件采用默认值兜底，Agent 在长任务里就会从那个薄弱点开始漂移。所以 Harness 工程的本质，是把所有默认值都从"框架自带"切换到"项目自己决定"——这件事，没有快捷方式，只能慢慢搭。

## 7. 仓库即事实源：因为 Agent 没有"口头传承"

从这一节开始，我们一条条挖 Harness 的核心原则。第一条原则也是最反直觉的一条——**代码仓库是 Agent 的唯一事实源（Single Source of Truth）**。听起来朴素，但它要求你改变持续多年的知识沉淀习惯，落地的时候会处处反直觉。

反直觉在哪里？人类团队的知识从来都不是只活在 git 里的。它散在 Slack 的某条置顶、Confluence 上半年前的设计文档、某次站会大家点头默认的共识、某个老员工脑子里"凡是动这一块都得问我"的隐规则。人类工程师之所以能 navigate 这一切，是因为他们活在一个有口头传承、有走廊偶遇、有"问问老张"的物理世界。**Agent 没有这些**——它的整个宇宙就是你给它读的那些文件，外加它能跑的那些工具。

这就意味着任何**没在 git 里的知识，对 Agent 来说在物理上不存在**。"动这个模块前要先 ping 一下 Bob"——Agent 不会 ping Bob。"这一段代码看起来冗余但其实是为了兼容三年前那个 incident"——Agent 不知道有这个 incident。"我们团队默认禁止用 lodash"——Agent 不会知道。它会按训练数据里最常见的方式去做，结果就是踩到你团队的每一颗地雷。

所以这条原则的具体动作是——**把所有"团队约定 / 业务术语 / 设计决策 / 反模式清单 / 历史教训"翻译成版本控制下的 Markdown 文件**。Hashimoto 的做法是给每个模块下放一份 NOTES.md 或 AGENTS.md，写清楚"这里有哪些不能动的边界、为什么这样设计、踩过哪些坑"。这些注解人类未必每次都看，但 Agent 会——而且看得很仔细，因为它没有别的事实来源。

这条原则还有一个隐藏红利——**它强迫团队把隐性知识显性化**。"动这块要先问 Bob"会被翻译成 "auth/AGENTS.md 写清楚 token 失效逻辑由 Bob 是 owner，PR 须 review"；"这段冗余代码兼容 incident 是因为..."会被翻译成代码注释或者 incident archive。一份资料，两份红利——新人和 Agent 都受益。这是为什么三家公司复盘里都强调："给 Agent 写 onboarding 文档的过程，意外地帮我们把团队 wiki 补齐了。"

## 8. AGENTS.md 分层：注意力的物理学

第一条原则要求"把知识写进仓库"，第二条原则解决"写进去之后怎么组织"。答案是 **AGENTS.md 分层目录**——根目录一份做总览，每个模块下放一份子文件。这件事 OpenAI、Anthropic、Hashimoto 三家独立得出相同结论，背后是同一条物理规律。

为什么不能一个大文件搞定？两个根本原因。

**第一个原因是注意力位置偏置（Position Bias）**。Transformer 的注意力机制虽然名义上是全局的，但实证研究反复发现——**模型对上下文开头和结尾的关注度，远高于中间**。这就是著名的 "Lost in the Middle" 效应。你把 5000 字的规则塞进一个 AGENTS.md，开头和结尾的规则能被模型记住，中间那部分基本等于白写。这不是模型不努力，是物理结构决定的。

**第二个原因是上下文成本（Token Cost）**。每次 Agent 跑任务都要把根 AGENTS.md 加载进窗口——5000 字 ≈ 5000 token，按当前 API 价格一次任务可能要烧掉好几分钱，更重要的是它**挤占了真正干活需要的工作记忆**。窗口里被规则占满，留给代码和工具结果的空间就少了。

正确做法是把 AGENTS.md 用成"目录页"——根目录的 AGENTS.md 只放最顶层信息（项目结构、5 条不可破规则、子模块指南位置），每个模块下放一份针对那块的细则。Agent 进 `src/auth/` 干活就自动加载 `src/auth/AGENTS.md`，离开那块时这部分上下文被自然丢掉。这是"**按任务局部性组织信息**"，恰好对齐了 Agent 的工作流。下面是一份可以直接拿去当模板的根 AGENTS.md：

```markdown
# Agents Guide

> 本文件是 AI Agent 在本仓库工作时的入口。先读完，再去具体模块。

## 不可破坏的硬规则（违反 = CI 红）
1. 不要 commit 任何 secret —— 只能改 `.env.example`
2. 所有 HTTP handler 必须经过 `internal/service/` 访问 DB
3. Public API 改动必须配 `tests/integration/` 测试
4. 数据库迁移走 `migrations/` 目录，必须有 rollback 脚本
5. 任何用 `lodash` 的代码 PR 会被自动 reject

## 模块导航
- `src/auth/AGENTS.md`     — Token、权限、刷新机制
- `src/billing/AGENTS.md`  — Stripe 集成、幂等、退款
- `src/inference/AGENTS.md` — 模型路由、超时、fallback

## 风格约定
- 命名 / 错误处理 / 日志：见 `docs/style.md`
- 提交信息：Conventional Commits
- 测试：每个模块的 AGENTS.md 各有要求

```

这种"目录 + 分层"结构能让 Agent 的注意力天然按任务聚焦——动 auth 就读 auth 的指南，动 billing 就读 billing 的；根目录因为短小精炼每次都加载也无压力。**规则的组织结构比规则本身更重要**，这是注意力物理学给我们上的一课。

## 9. Linter 是 Agent 的实时教练：反馈环路设计

第三条原则是 OpenAI 那篇复盘里花了最大篇幅讲的事——**把所有可被机器检查的规则，都写成自定义 Linter**。这条原则的真正深度，不在"加了 Linter"这件事本身，而在于**它把人和 Agent 之间的反馈环路从离线异步变成了在线实时**。

传统 Linter 服务的对象是人类。报错信息长这样："Error: importing from internal/db is not allowed in handlers/."——简短、机器化、留给人类判断怎么改。人类看到这条会去 Slack 问"为啥不让这么 import？"或者翻 wiki 查规则历史。整个反馈环路的延迟以小时计，**反馈里没有"修复指引"，只有"违规判定"**。

当把 Linter 的服务对象换成 Agent 之后，这个设计就严重失配——Agent 没法去 Slack 问 Bob，它**只能根据它眼前看到的报错信息自我修正**。如果报错只说"不允许"不说"为什么 + 怎么改"，Agent 会瞎猜：要么粗暴地把整个 import 删了再重写、要么绕过规则换个魔法路径、要么干脆把代码注释掉。每一种猜测都是新的漂移。

OpenAI 的解法是把 Linter 报错**升级成"小型教程"**——每条规则的报错里同时包含"为什么这条规则存在"和"正确的做法是什么 + 一个可照抄的示例"。看下面这对样例，第一眼就能感受到差距：


> **传统 Linter（给人看）**
> Error: handlers/user.go:42  
> Direct import from \`internal/db\` is not allowed in HTTP handlers.


> **Harness Linter（给 Agent 看）**
> Error: handlers/user.go:42  
>
> Direct import from \`internal/db\` is not allowed in HTTP handlers.  
>
> **Why this rule exists:**  
> Handlers must not bypass the service layer, which handles auth, rate limiting, and audit logging. Direct DB access from handlers caused incident #2024-08-17.  
>
> **How to fix:**  
> Move the query into a method on \`internal/service/user.UserService\`, then call it from the handler. See \`handlers/order.go\` for a working example.


Agent 看到第二种报错的反应不是瞎猜，而是**立刻去读 handlers/order.go、抄一份结构、改回去、重新提交**。整个修复发生在一次 tool call 之内，不需要人类介入。这件事的意义远超 "Linter 写得更详细一点"——**它把 Agent 从"被人审"的客体变成了"自我审、自我改"的主体**。反馈环路从离线小时级变成了在线秒级，纠错效率提升了几个数量级。

这件事推广到整个工程实践上，会产生一个深远的认知转变——**报错信息、提示信息、CI 输出、test 失败信息，全部都是写给 Agent 看的"上下文"**，而不是写给人类调试时看的标签。一个 Harness 工程做得好的团队，会把所有这些"反馈面"统一升级成 Agent 友好的格式。这个升级不光让 Agent 更稳，附带还让人类新员工的体验也变得无比丝滑——又是一份资料，两份红利。

## 10. 用约束换自主：高速公路悖论

到这里你应该已经感受到一个反直觉的味道——**Harness 工程里，"加约束"和"给自由"居然是同义词**。这一节专门把这个悖论拆开讲，因为它是整套体系最违背朴素直觉、却又最深刻的一条原则。

朴素直觉认为：约束越多，自主越少。这在物理世界里大体成立——你给一个人戴上更多脚镣，他能做的事情就越少。但在**智能体行为里，这个关系会反转**。先用一个生活例子打通：你给一个刚拿驾照的新司机两条路选——

- **路 A：山区小路，没有车道线、没有限速、没有护栏，全靠你自己判断**。
- **路 B：标准高速公路，有清晰车道线、限速牌、护栏、紧急停车带**。

哪条路上新司机敢开快？显然是路 B。**不是因为路 B "自由度更高"，而是因为它的"约束"明确到让司机不需要每秒钟评估"我这一脚踩下去会不会出事"**。约束承担了认知负担，于是司机有空间去专注真正需要判断的事情——超车时机、出口选择、车速调节。

Agent 的行为完全是同一回事。**明确的规则让 Agent 不需要在每一步重新评估"我能不能这么干"**。它写一个 handler 时不需要纠结"我直接 import db 包行不行"——因为 Linter 会立刻拦截，硬约束在那里。它写完测试不需要纠结"测试该放哪个目录"——因为 AGENTS.md 写明了。它**把节省下来的认知带宽全部投入到真正需要智能的地方**——业务逻辑该怎么实现、边界条件该怎么处理、性能瓶颈在哪儿。这就是它"看起来更自主"的源头。

反过来，规则模糊的项目里 Agent 看起来反而**更不自由**——它会在每一步反复试探边界，要么过度保守什么都不敢做、要么偶尔大胆一回踩雷。这种"无规则导致的迟疑"在长任务里会指数级放大，最终表现为漂移甚至完全卡死。

这条原则在更高层还有几个等价表述，都是同一个道理的不同侧面：

- **静态类型系统不是限制是赋能**——TypeScript 的类型让你敢重构、敢删代码，因为你知道编译器会兜底。
- **"让非法状态不可表达"是好设计**——Rust 借用检查器、可辨识联合类型，都是把错误用法编码进类型系统，让你**必须**写对。
- **法律不是限制自由是创造自由**——清晰的产权、合同、法治，反而让市场参与者敢于做长期投资。

把这条原则放到 Harness 工程里，就有一个非常具体的工作哲学——**所有能编码的决策都编码下来，把剩下来真正模糊的部分留给模型判断**。这不是降低 Agent 的"智能上限"，而是把它的智能集中到真正需要的地方。这条原则在哪里都管用，但在 Agent 介质上效果最戏剧化——因为模型的"判断成本"和"判断风险"都比人类高，把可编码的决策固化能省下大量两者。

## 11. 执行流程：从无界探索到有界搜索

第四件套是执行流程。这一层管的是 Agent 主循环的**形状**——它什么时候允许往前一步、什么时候必须停、什么时候要反思。绝大多数 Agent 漂移问题，最终都能追溯到执行流程没有显式设计、靠 while(true) 兜底。

裸的 Agent 主循环长这个样子——读上下文、决定下一步、调工具、把结果塞回上下文、回到读上下文。这是一个**没有出口、没有节奏控制、没有反思机制的无界循环**。理论上它会一直跑到模型说"我做完了"或者撞到 token 上限。问题是：长任务里模型经常自己都不知道"做完了没有"，一旦它在某一步陷入局部最优（比如反复改同一行代码），整个循环就在那打转，外面看就是经典的"卡住"。

Harness 化的执行流程，会在这个朴素循环里嵌几个关键控制点。**最基础的是 step limit**——给单次任务设个最大 tool call 数（比如 50），跑到上限强制停下来让用户介入。这个看似简单的限制能挡掉至少一半的"无限自我修正死循环"。**第二个是 checkpoint 机制**——每 N 步或者每完成一个里程碑，让 Agent 把"我做到哪了 / 接下来要做什么"写成 summary 提交到一个固定文件里；下一步读这个文件作为锚点，相当于给长任务装了"中途存档"。**第三个是显式反思（self-reflection）**——在关键节点强制 Agent 跳出执行模式，先回答几个 meta 问题（"我有没有偏离原任务？我有没有重复劳动？是不是该停下来问用户？"）再继续。

把这三件事加进去之后，Agent 的执行从"无界探索"变成了**"有界搜索"**——它依然有大量自由度去探索具体怎么实现，但漂移到极远的概率被严重压低。这跟传统软件里 timeout、circuit breaker、retry with backoff 是同一个谱系的设计——**都是给一个本来可能跑飞的进程装上"知道什么时候该停"的能力**。

这一层最容易被忽视，因为它"什么都不做的时候"看不见效果。但一旦 Agent 跑了一个跨小时的复杂任务，没有执行流程兜底的体验和有兜底的体验，差距巨大——前者是"我得整天盯着、随时介入"，后者是"我可以放心去开个会，回来看交接 summary"。

## 12. 上下文管理：窗口经营是一门手艺

第五件套是上下文管理。把它放到最后讲，是因为它最像"手艺活"——没有银弹，全是**权衡**。它管的是这件事：模型每一步看到的窗口里，到底装什么、不装什么、装多久。

先纠正一个常见误解——**上下文窗口不是内存**。内存的语义是"你写进去的东西一直在那等你读"。窗口的语义是**"每一次推理的 tabula rasa——你这次塞进去什么，模型这次就能看到什么；不塞，就看不到"**。它更像一张工作台：每次干活前你要重新摆，干完一步要重新整理。

窗口经营的核心矛盾是：**窗口大小是有限的（哪怕 1M token，相对长任务也终会爆），但任务需要的信息会随时间积累**。所以你不可避免要做"取舍"——决定什么留下、什么扔掉、什么压缩。常见的几种经营手段：

- **Compaction（压缩）**——把多轮历史交互压缩成一段 summary，腾出空间放新内容。代价是细节丢失，benefit 是窗口续命。
- **Eviction（驱逐）**——直接扔掉久远的、不相关的早期内容。比 compaction 更激进，适合那些"早期讨论已经定盘、不会再回头"的任务。
- **Memory promotion（记忆提升）**——把窗口里反复用到的关键事实写到外部记忆文件，下次按需召回，而不是占着窗口位置。这是 Letta、MemGPT 那类"长记忆"系统的核心思路。
- **Just-in-time loading（按需加载）**——AGENTS.md 那种分层结构本质就是这个——进哪个目录加载哪份文档，离开就丢，窗口里始终只有当前任务相关的资料。

什么时候用哪种，没有标准答案，看任务形态。重构任务里"早期文件已经改完"的内容可以激进 eviction；调研任务里所有发现都可能有用，更适合 compaction；多日跨会话的任务，必须搭 memory promotion。**Harness 工程师的一大半时间，其实就是在调这些参数和策略**——这就是为什么我说它是手艺活。

不过手艺背后还是有个底层原则可以记——**窗口里的每一个 token 都应该为"当前这一步的判断"服务，不为这一步服务的就该挪走**。这跟数据库里"工作集 vs 冷数据"、操作系统里"驻留页 vs 换出页"是同一套思想。Harness 工程在这一层借的，正是计算机体系结构积累了几十年的内存层级管理智慧。

## 13. 从 0 到 1 搭一个 Harness（实操路线图）

讲了 12 节原理，落到一个具体团队头上到底要怎么开始？这里给一份"两周走一步、三个月成型"的可执行路线图。它不是教条，但能帮你避开前两个月最常见的坑。

**第 1 步（第 0 周）：跑通"裸 Agent"，建立 baseline**。不要急着搭 Harness。挑一个真实任务（修一个非平凡的 bug、加一个跨模块的小功能），让 Claude Code / Codex / Cursor 这类工具裸跑，**认真观察它在哪些点稳、哪些点崩**。把崩溃样本一条条记下来——这些就是后面 Harness 要打磨的尖刺低谷。没有这步 baseline，所有改造都是凭想象。

**第 2 步（第 1-2 周）：建根 AGENTS.md**。把团队里"老员工才知道的隐性知识"写下来。重点不是穷举所有规矩，而是先列 **5 条不可破的硬规则**。这五条要短到 Agent 一定会读、一定会记住。其它细节先放到 TODO。

**第 3 步（第 3-4 周）：把第一条规则机械化**。从 5 条硬规则里挑最容易写成 Linter 的一条（通常是 import 限制或者 naming 约定），实现成 pre-commit 钩子或 CI 检查。然后把**报错信息按"为什么 + 怎么改 + 示例"的格式重写**。这一步效果立竿见影——你会亲眼看到 Agent 第一次"自我修复"。

**第 4 步（第 5-6 周）：按模块拆 AGENTS.md**。当根目录的 AGENTS.md 超过 1500 字以后，就该往子目录拆。给每个核心模块下放一份子文件，根目录的那份缩回纯目录。这一步对长任务稳定性是阶跃式提升，因为它解决了"中间遗忘"。

**第 5 步（第 7-8 周）：搭中间件钩子**。挑几个 baseline 阶段最痛的崩溃模式，对应加钩子。常见组合：pre-commit 跑 Linter、post-tool-call 截断超长输出、context-trim 在窗口快满时 summarize、step-limit 限制单任务 tool call 上限。**不要为不存在的问题加钩子**，每个钩子都应该对应你 baseline 里观察到的具体崩溃。

**第 6 步（第 9 周开始持续做）：每周 Agent 复盘**。挑几个本周 Agent 跑过的真实任务，看它在哪里走偏、人类怎么救回来的。把这次"救援经验"翻译成新的 AGENTS.md 条目、新的 Linter 规则、新的钩子。**Harness 是长期工程，没有"做完"的那一天，只有"今天比昨天稳一点"**。

> **推荐节奏：**按这条路线图走，三个月内能搭出比"裸 Agent"稳定 5-10 倍的 Harness 体系。这不是空口数字，是 OpenAI、YC 几家 Agent 创业公司复盘里反复出现的体感。比"一次到位写完所有规则"更扎实，也更符合迭代式工程的本质。

## 14. Harness 工程师：一个新角色正在浮现

最后聊一聊"人"。Harness 工程不是某个现有岗位的副业——它正在催生一个新角色：**Harness 工程师**。这个角色处于 ML 工程师和后端工程师之间，但又都不完全是。

类比一下，会更清楚这个新角色长什么样。DevOps 工程师是 2010 年前后浮现的——它处于"运维"和"开发"之间，既懂应用又懂基础设施，本质是**把两个原本割裂的领域用工程化方法粘起来**。Site Reliability Engineer 是 Google 命名的另一面，本质相同。今天的 Harness 工程师扮演同样的"夹缝角色"——它处于"模型"和"业务系统"之间，既懂模型的脾气（spiky、attention bias、context window 经济学）又懂工程系统的设计（Linter、CI、中间件、状态机），本质是**把概率性的模型行为塞进确定性的工程框架里**。

未来 3-5 年，几件事很可能会发生：

- **Harness 框架会标准化**——就像今天的 React Router、Express middleware、Kubernetes operator 一样，会有几个事实标准的 Harness 框架出现，承担"五件套"的脚手架工作。你不需要每次都从零搭。
- **但具体的 Harness 内容会永远项目特化**——AGENTS.md 写什么、Linter 设什么规则、钩子拦截什么，这些永远绑在你公司业务上，没法买现成的。
- **"Harness as a Service"可能出现**——专门给中小团队提供托管 Harness 的 SaaS，把通用部分（Linter 引擎、上下文管理、执行流程编排）做成产品，让团队只需要往里写业务规则。
- **Harness 工程会成为简历加分项**——就像今天会写 Dockerfile / 懂 Kubernetes 是加分项一样。"我搭过一个 Coding Agent Harness、把团队 Agent 漂移率从 X 压到 Y"会成为面试里常被问的故事。

更深层的转变是工程师的**心智模型**。过去十年我们习惯了"我直接写代码"。再往前几年我们也曾习惯过"我直接拉电缆、装机器"。每一代基础设施的成熟，都让上一代工作被自动化掉，让工程师爬到更高的抽象层。**Harness 工程是下一层抽象**——你不再直接写每一行业务代码，而是设计一个让 Agent 能稳定写出业务代码的环境。这听起来像在"放手"，本质上是在**把判断力投入到更高杠杆的位置**。

> 文章到这里结束。回到最开始那个场景——你看着 Agent 跑到一半开始漂移，拍着桌子在想"是模型不行还是我用法不行"。读完这一篇之后，希望你下次面对同样画面时，能立刻分辨出来：是 Prompt 没钉死、还是 Context 在稀释、还是 Harness 缺了某一片。然后挽起袖子，把那片缺口补上——这就是 Harness 工程的全部艺术。
