团队博客最常见的问题不是没人会写，而是写作被当成额外任务。真正可持续的团队内容系统，需要把选题、资料、评审和发布节奏变成一条轻量但稳定的工作流。

<Callout tone="accent">
第一版系统不追求复杂 CMS。只要本地文件结构清楚、元数据完整、发布命令稳定，就足够支撑一个高质量团队博客。
</Callout>

## 选题要从资产视角出发

如果每篇文章都只服务当周热点，团队会很快失去耐心。更好的方式是把文章看成可复用资产：新人培训、销售答疑、产品解释、工程复盘、招聘展示，都可以从同一篇内容里获益。

一个简单的选题评估可以包含三项：

- 是否能长期回答一个重复问题
- 是否能展示团队的独特判断
- 是否能被内部流程持续引用

## 元数据决定内容能否被组织

本项目把文章放在 `content/posts` 下，每篇文章通过 frontmatter 声明标题、描述、日期、分类和标签。这样列表页、标签页、RSS 和 Sitemap 都能从同一份数据自动生成。

```ts
export type PostMeta = {
  title: string;
  description: string;
  date: string;
  category: string;
  tags: string[];
};
```

## 评审不是润色，而是降低理解成本

团队文章的评审重点不是把风格统一成一种声音，而是保证读者能迅速理解上下文、判断结论可靠性，并知道下一步该做什么。好的评审会删掉炫技表达，补足背景和证据。

## 发布节奏要小而稳定

每月两篇高质量文章，通常比短期密集发布更有价值。稳定节奏会倒逼团队积累素材，而不是在发布前临时拼凑。
